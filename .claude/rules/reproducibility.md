# Reproducibility & run contract

This file is the **single source of truth** for what a run records, what a
build records, and how the two are linked. The `reproducibility-check` hook
parses this file's schema definitions; do not paraphrase fields elsewhere.

The same file covers four concerns that codex's review flagged as easy to
fragment across multiple rules:

1. **Run metadata schema** (per-execution_target).
2. **Build provenance** (`build_id` ↔ source ↔ toolchain ↔ binary).
3. **HPC scheduler capture** (cluster runs only).
4. **Config snapshots** (per-run immutable copy of the experiment config).

Safety constraints for `device` / `hil` runs live in `safety-hil.md`.

---

## 1. Run directory layout

Every run is identified by a `run_id` of the form
`<UTC ISO-8601 with hyphens>_<8-char hash>` (e.g.
`2026-05-14T12-34-56_a1b2c3d4`). The directory under `data/results/<run_id>/`
contains:

```
data/results/<run_id>/
  metadata.json          REQUIRED — see §2
  config.snapshot.yaml   REQUIRED — immutable copy of the resolved experiment config
  log.txt                REQUIRED — combined stdout/stderr of the run
  results/               raw results (CSV/Parquet/NPZ/binary checkpoints/…)
  figures/               optional, generated by /analyze-results
  telemetry/             device|hil only — raw device traces, sensor streams
  cleanup_log.txt        device|hil only — what the safe-stop / abort path did
```

`results/` and `telemetry/` are bag-of-files; the schema is experiment-specific
and described in the experiment's own `docs/research/methodology.md`.

---

## 2. metadata.json schema

The schema is **nested by `execution_target`**. The enum is
`sim | device | hil` — there is no `mixed` value at run scope. A workflow
that combines simulation precompute with a later device playback is modeled
as two distinct runs (one `sim`, one `device`) that share an
`experiment_family`.

Top-level fields are required for every run. The conditional blocks are:

| Block | Required when |
|---|---|
| `sim` | `execution_target == sim` |
| `device` | `execution_target == device` OR `execution_target == hil` |
| `hil` | `execution_target == hil` |
| `scheduler` | `compute_target == cluster` |

There is **no `build` block** in run metadata. Native builds have their own
immutable record under `data/builds/<build_id>/manifest.json` (§3). A run's
top-level `build_id` field is the cross-reference; tools that need build
provenance read the manifest by id.

A `hil` run carries both a `device` block (describing the device-under-test)
AND a `hil` block (describing the HIL bench and coupling). It does NOT carry
a `sim` block — the simulator side of a HIL loop is recorded in the `hil`
block.

```json
{
  "run_id": "2026-05-14T12-34-56_a1b2c3d4",
  "experiment_id": "solver-baseline",
  "experiment_family": "solver-accuracy",
  "sweep_id": null,
  "execution_target": "sim",

  "started_at": "2026-05-14T12:34:56Z",
  "finished_at": "2026-05-14T12:36:01Z",
  "exit_state": "success",
  "pid": 81234,

  "git_rev": "<commit-sha-or-dirty>",
  "git_clean": true,
  "script": "src/experiments/solver-baseline/run.py",
  "entrypoint_sha256": "<sha-256-of-script-file>",
  "args": {"...": "..."},
  "config_snapshot_path": "config.snapshot.yaml",
  "config_snapshot_sha256": "<sha-256-of-snapshot-file>",
  "build_id": "<id-or-null>",

  "compute_target": "local",
  "platform": "darwin-arm64",

  "sim":       { /* see §2.2 */ },
  "device":    { /* see §2.3 */ },
  "hil":       { /* see §2.4 */ },
  "scheduler": { /* see §2.5 */ }
}
```

### 2.1 Top-level required keys (all runs)

| Key | Type | Notes |
|---|---|---|
| `run_id` | string | matches directory name |
| `experiment_id` | string | resolves to Zone B `experiments:` entry |
| `experiment_family` | string \| null | for cross-run comparison |
| `sweep_id` | string \| null | non-null when run is part of a sweep |
| `execution_target` | enum | `sim` \| `device` \| `hil` |
| `started_at` | ISO-8601 UTC | wrapper writes before launching |
| `finished_at` | ISO-8601 UTC \| null | null until written; `null` is allowed on `crash` / `interrupted` runs caught by the orphan-run reaper |
| `exit_state` | enum \| null | `success` \| `crash` \| `aborted` \| `safe-stop` \| `interrupted`. May be `null` between `write_initial_metadata` and `finalize_metadata` (i.e. while the run is in flight). The orphan-run reaper distinguishes "in flight" from "interrupted" by inspecting `finished_at` and the recorded `pid`, NOT by reading a special `running` value. |
| `pid` | int | OS pid of the run wrapper process. Used by `session-start.py` to distinguish a still-running orphan from a truly interrupted one. Written by `experiment-runner` via `repro.write_initial_metadata` at launch (`os.getpid()`). |
| `git_rev` | string | commit sha; suffix `-dirty` if working tree dirty |
| `git_clean` | bool | redundant with `-dirty` suffix but cheap to validate |
| `script` | string | path of the entrypoint relative to repo root |
| `entrypoint_sha256` | string | SHA-256 of the entrypoint file at launch time; used by safety-hil cleanup-handler trust check |
| `args` | object | full CLI args / launcher args |
| `config_snapshot_path` | string | relative to `run_id` dir, normally `config.snapshot.yaml` |
| `config_snapshot_sha256` | string | SHA-256 of the config snapshot file; used by safety-hil dry-run equivalence check |
| `build_id` | string \| null | required when resolved `runtime != python-uv`; see §3 |
| `compute_target` | enum | `local` \| `cluster` \| `device` \| `hil` |
| `platform` | string | `<sysname>-<arch>`, e.g. `darwin-arm64`, `linux-x86_64` |

### 2.2 `sim` block (required when `execution_target == sim`)

| Key | Required | Notes |
|---|---|---|
| `seed` | yes | int; per-framework seeds in `seeds_per_framework` |
| `seeds_per_framework` | optional | e.g. `{numpy: 42, torch: 42, jax: 42}` |
| `python_version` | yes when runtime touches Python | e.g. `3.12.4` |
| `package_versions` | yes when runtime touches Python | full `uv pip freeze` snapshot |
| `mpi_runtime` | required when MPI is used | e.g. `OpenMPI 5.0.5` |
| `solver_version` | optional | for native solvers, the internal version string |
| `gpu` | optional | string description; null for CPU-only |
| `cuda_version` | optional | required if GPU code paths ran |
| `determinism_caveats` | optional | free-text; non-empty when nondet ops were used |

### 2.3 `device` block (required when `execution_target ∈ {device, hil}`)

| Key | Required | Notes |
|---|---|---|
| `device_id` | yes | matches the lock file name; unique device identifier |
| `device_model` | yes | vendor + model |
| `firmware_rev` | yes | revision string from the device |
| `calibration_ref` | yes | filename or hash under `data/calibrations/` |
| `calibration_age_h` | yes | hours since the calibration was performed; safety-check uses this |
| `ambient` | recommended | `{temperature_c, humidity_pct, supply_voltage_v, ...}` as relevant |
| `operator` | yes when `safety_class != none` | human operator name or initials (string) |
| `lock_path` | yes | path to `data/locks/<device_id>.lock` held during the run |
| `operator_confirmation_at` | yes when `safety_class == destructive` | ISO-8601 UTC of the operator-in-loop confirmation; see safety-hil.md §1 |
| `dry_run_rehearsal_run_id` | yes when `safety_class ∈ {calibration-required, destructive}` and `dry_run != true` | the `run_id` of the successful dry-run rehearsal whose `config_snapshot_sha256` matches this run |
| `safety_overrides` | list | objects: `{check_name, reason, override_at, operator}` — populated when `--override-safety` is used |
| `dry_run` | bool | true when this run itself is a dry-run rehearsal (no actuation) |

### 2.4 `hil` block (required when `execution_target == hil`)

A HIL run includes BOTH a `device` block (the device-under-test) AND this
`hil` block (the bench and its coupling). The simulator side of the HIL loop
is described inside this block (not in a separate `sim` block).

| Key | Required | Notes |
|---|---|---|
| `bench_id` | yes | HIL bench identifier; lock under `data/locks/<bench_id>.lock` |
| `bench_lock_path` | yes | path to `data/locks/<bench_id>.lock` held during the run |
| `coupling_mode` | yes | enum: `closed-loop` \| `open-loop` \| `record-replay` |
| `sample_rate_hz` | yes | bench sample rate |
| `interlocks` | yes | object: `{e_stop_armed: bool, watchdog_ms_max: int, watchdog_ms_used: int, ...}` |
| `bench_selfcheck_path` | yes | path to `data/locks/<bench_id>.selfcheck.json`; safety-hil requires this within the last hour |
| `simulator` | yes | object describing the simulator side of the loop: `{kind, version, integration_step_s, ...}` |

### 2.5 `scheduler` block (required when resolved `compute_target == cluster`)

| Key | Required | Notes |
|---|---|---|
| `kind` | yes | `slurm` \| `pbs` \| `lsf` \| `sge` \| `kubernetes` |
| `job_id` | yes | scheduler-assigned job id |
| `queue` | yes | queue / partition name |
| `node_type` | yes | node SKU or feature string |
| `nodes` | yes | count |
| `cores_per_node` | yes | int |
| `gpus_per_node` | optional | int or null |
| `walltime_requested_s` | yes | int seconds requested |
| `walltime_used_s` | optional | int seconds actually used; null on aborted/interrupted |
| `job_script_hash` | yes | SHA-256 of the submitted job script |

## 3. Build provenance

Native runs (`runtime ∈ {cpp-cmake, rust-cargo, make}`) MUST produce a build
artifact recorded under `data/builds/<build_id>/`. Pure Python runs MAY skip
this. The `build_id` is referenced from every run that consumed that build.

### 3.1 `build_id`

`<UTC ISO-8601 with hyphens>_<8-char hash>` of the source tree plus toolchain
identifiers. The hash is computed by `build-engineer` over: `git_rev`,
toolchain strings, build flags, container image digest (if any). Two builds
with identical inputs produce the same `build_id` (it is content-addressable
modulo timestamp prefix).

### 3.2 `data/builds/<build_id>/manifest.json`

```json
{
  "build_id": "2026-05-14T12-30-00_e5f6a7b8",
  "started_at": "2026-05-14T12:29:30Z",
  "finished_at": "2026-05-14T12:30:00Z",

  "runtime": "cpp-cmake",
  "git_rev": "<sha>",
  "git_clean": true,
  "source_tree_hash": "<sha-256-of-tracked-files>",

  "toolchain": {
    "compiler": "gcc",
    "compiler_version": "13.2.0",
    "compiler_path": "/usr/bin/gcc",
    "linker": "ld",
    "linker_version": "2.41",
    "build_flags": ["-O3", "-march=native", "..."],
    "cmake_version": "3.28.1",
    "cmake_args": ["...", "..."],
    "abi": "x86_64-linux-gnu"
  },

  "container": {
    "image": "ghcr.io/...",
    "digest": "sha256:...",
    "engine": "docker"
  },

  "mpi": {
    "runtime": "OpenMPI",
    "version": "5.0.5"
  },

  "artifacts": [
    {
      "path": "build/bin/solver",
      "kind": "executable",
      "sha256": "...",
      "size_bytes": 12345678
    }
  ],

  "platform": "linux-x86_64",

  "exit_state": "success",
  "smoke_test": {
    "command": "./build/bin/solver --version",
    "exit_code": 0,
    "stdout_tail": "solver 1.2.0",
    "duration_s": 0.04
  }
}
```

`toolchain`, `mpi`, `container` blocks are conditionally required: include
those that were actually used.

Manifest `exit_state` enum: `success` (compile OK, smoke test OK) | `smoke_failed` (compile OK, smoke test failed) | `failed` (compile failed; partial manifest written so the failure is traceable).

`smoke_test` is required when `exit_state ∈ {success, smoke_failed}` and absent for `failed`.

### 3.3 Reuse rules

- A run MAY reference an existing `build_id` instead of rebuilding. The
  `experiment-runner` agent verifies (a) the manifest exists, (b) the
  referenced artifact hash matches the actual file on disk, before launching.
- A rebuild that produces identical inputs SHOULD reuse the same `build_id`
  (the hash matches). If two builds produce different `build_id` for what the
  user thinks is the same source, `build-engineer` reports the diff (likely a
  different toolchain or container image).

## 4. Config snapshots

Every experiment has a config (YAML or JSON). On launch:

1. The `experiment-runner` agent resolves the config (merges defaults, CLI
   overrides, environment overrides).
2. Validates against the experiment's schema (`config_schema` in Zone B).
3. Writes the resolved config verbatim to
   `data/results/<run_id>/config.snapshot.yaml`.
4. Records `config_snapshot_path` in `metadata.json`.

The snapshot is immutable. A re-run of the same config produces a new
`run_id` with an identical (byte-for-byte) snapshot — this is the canonical
way to verify reproducibility.

---

## 5. Seed policy

- Set seeds for: Python `random`, `numpy`, framework RNGs (torch, jax, tf, etc.).
  For C++ / Rust runs, the seed is passed through the config and consumed by
  the entrypoint.
- The seed is recorded in `sim.seed`. Per-framework seeds, if they differ from
  the master seed, go in `sim.seeds_per_framework`.
- Multiple seeds for replication are listed in `docs/research/methodology.md`.
  Each seed produces a separate `run_id`; they MAY share a `sweep_id`.

---

## 6. Determinism caveats

- If full determinism is impossible (cuDNN nondeterministic kernels, race
  conditions in MPI reductions, etc.), record this in `sim.determinism_caveats`
  AND mention it in `docs/research/methodology.md`. Report variance across at
  least 3 seeds in that case.

---

## 7. Data provenance

- Every file under `data/raw/` should have a `data/raw/<name>.README.md`
  describing source, license, retrieval date, and any preprocessing already
  applied at source.
- `data/processed/` is rebuilt by a script in `src/`, never manually.
- `data/results/<run_id>/` is immutable once written (see
  `research-integrity.md` §Data handling).
- `data/builds/<build_id>/` is immutable once written. If a manifest's
  recorded artifact hashes do not match the files on disk, the build is
  considered corrupt and `build-engineer` MUST rebuild.

---

## 8. Orphan-run reaper

When a process is killed -9 or the host loses power mid-run, the wrapper
cannot write `finished_at` or `exit_state`. On next session start,
`session-start.py` scans `data/results/*/metadata.json` for entries with
`finished_at == null` AND last-modified-time older than a grace period
(default 1 hour) AND no PID in `ps`. Such runs are patched to:

```json
{ "exit_state": "interrupted", "finished_at": "<scan_time>", "reaper": true }
```

The reaper never deletes data; it only patches the metadata so analysis
scripts can skip these runs cleanly.

## 9. Writer responsibility (single source of truth)

Every metadata field has exactly one party that writes it. The
`reproducibility-check` hook is a validator, not a writer. The same applies
to build manifests under `data/builds/<build_id>/`. The table below is
authoritative; agent and skill docs that contradict it are wrong.

### 9.1 Run metadata writers

| Field | Writer | When |
|---|---|---|
| `run_id`, `experiment_id`, `experiment_family`, `sweep_id`, `execution_target`, `started_at`, `git_rev`, `git_clean`, `script`, `entrypoint_sha256`, `args`, `config_snapshot_path`, `config_snapshot_sha256`, `build_id`, `compute_target`, `platform`, `pid` | `experiment-runner` via `repro.write_initial_metadata` | At launch, before delegating to the driver. |
| `finished_at`, `exit_state` | `experiment-runner` via `repro.finalize_metadata` | On every exit path (atexit + signal handlers). Idempotent. |
| `sim.seed`, `sim.seeds_per_framework`, `sim.python_version`, `sim.package_versions`, `sim.gpu`, `sim.cuda_version`, `sim.determinism_caveats` | `experiment-runner` via `repro.write_initial_metadata` (extends `sim` block automatically when `execution_target == sim`) | At launch. `gpu` / `cuda_version` are populated when the driver detects GPU usage. |
| `sim.mpi_runtime`, `sim.solver_version` | `experiment-runner` via `repro.patch_metadata` | After the native binary reports its version on first invocation, or when MPI is initialized. |
| `device.device_id`, `device.lock_path` | `experiment-runner` via `repro.write_initial_metadata` (read from Zone B registry + lock file) | At launch. |
| `device.device_model`, `device.firmware_rev`, `device.calibration_ref`, `device.calibration_age_h`, `device.ambient`, `device.operator` | `device-operator` via `repro.patch_metadata` | After device init (`device_model`, `firmware_rev`) and after calibration freshness check (`calibration_ref`, `calibration_age_h`). `ambient` is sampled at run start. `operator` is read from the lock file's `operator` field. |
| `device.operator_confirmation_at`, `device.dry_run_rehearsal_run_id`, `device.dry_run`, `device.safety_overrides` | `/run-experiment` skill via `repro.patch_metadata` | During §8 / §9 preflight; written BEFORE the driver is launched so the record is permanent even if the driver crashes immediately. |
| `hil.bench_id`, `hil.bench_lock_path` | `experiment-runner` via `repro.write_initial_metadata` (read from Zone B registry + lock file) | At launch. |
| `hil.bench_selfcheck_path`, `hil.interlocks` | `device-operator` via `repro.patch_metadata` | After the bench self-check runs (or is re-confirmed by `/run-experiment` §8.3). `interlocks` is copied from `<bench_id>.selfcheck.json`. |
| `hil.coupling_mode`, `hil.sample_rate_hz`, `hil.simulator` | `experiment-runner` via `repro.write_initial_metadata` (read from the resolved experiment config snapshot — these are config keys validated by `config_schema`) | At launch. |
| `scheduler.kind`, `scheduler.job_id`, `scheduler.queue`, `scheduler.node_type`, `scheduler.nodes`, `scheduler.cores_per_node`, `scheduler.gpus_per_node`, `scheduler.walltime_requested_s` | `experiment-runner` via `repro.write_initial_metadata` (read from scheduler env vars per `scheduler.kind`: SLURM_JOB_ID, SLURM_JOB_PARTITION, SLURM_JOB_NUM_NODES, etc.) | At launch on a cluster node. |
| `scheduler.job_script_hash` | `/run-experiment` skill (computed and passed through environment to the cluster node) OR `experiment-runner` if launched from inside an already-submitted job | At job submission time / at launch. |
| `scheduler.walltime_used_s` | `experiment-runner` via `repro.finalize_metadata` | On exit. |

### 9.2 Build manifest writers

All fields under `data/builds/<build_id>/manifest.json` (§3.2) are written by
`build-engineer` at build time. The manifest is immutable once written. The
`reproducibility-check` hook validates schema completeness but never modifies
the manifest.

### 9.3 Helper function contract

`src/utils/repro.py` exposes six **public** functions (stable API) plus two
**secondary read helpers**. The full signatures live in
`.claude/agents/experiment-runner.md` §4. The names are part of the public
template surface; do not rename them in agents or skills.

Public stable API:
- `make_run_id()` — allocate a `<UTC>_<8-char-hash>` run id
- `set_seed(seed, frameworks=None)` — set seeds for random / numpy / torch / jax / tf as applicable
- `hash_file(path)` — SHA-256 of a file
- `write_initial_metadata(run_id, *, experiment_entry, resolved, args, config_snapshot_path, build_id)` — write the full schema-complete `metadata.json` at launch (including `pid = os.getpid()`)
- `patch_metadata(run_id, **fields)` — atomic merge of mid-run updates; nested dicts merge shallow
- `finalize_metadata(run_id, exit_state)` — write `finished_at` + `exit_state` + `scheduler.walltime_used_s` when applicable; idempotent

Secondary helpers (used by native launchers and by `/run-experiment` step 5):
- `read_build_manifest(build_id)` — load `data/builds/<build_id>/manifest.json` as a dict
- `verify_binary_hash(manifest, artifact_path)` — raise on hash mismatch
- `compute_source_tree_hash()` — SHA-256 over the sorted list of tracked source paths + their file hashes. Independent of `git rev-parse HEAD` — a dirty working tree yields a different source-tree hash even at the same commit. Used for `build_id` cache lookup.

### 9.4 Reaper writers

`session-start.py`'s orphan-run reaper writes `exit_state: interrupted`,
`finished_at: <scan_time>`, `reaper: true` for runs that lost their wrapper
process before `finalize_metadata` could run. The reaper never writes any
other field; in particular, it does not invent `device.*` or `hil.*` data.
